# Notes on Local LLM Serving Framework Tradeoffs

The local LLM serving landscape is dominated by a few key tradeoffs between ease of use, hardware support, and performance. **llama.cpp** is the foundational engine powering many other tools, offering the broadest hardware support (including first-class Apple Silicon via Metal) and the most granular quantization options (Q2 through Q8, plus IQ and MXFP4 variants), making it ideal for resource-constrained environments. **Ollama** wraps llama.cpp in a user-friendly Go-based package manager, simplifying model downloads and serving with minimal configuration, though it inherits llama.cpp's quantization limits. **MLX/MLX-LM** is Apple's own framework, delivering excellent performance on Apple Silicon with a clean Python API, but it is restricted to Apple hardware and lacks built-in tool calling. **llamafile** takes a unique approach by bundling llama.cpp into a single self-contained executable, making distribution trivial, while **vLLM** and **SGLang** are high-throughput serving engines optimized for GPU clusters (primarily NVIDIA), with vLLM offering the widest quantization support and SGLang excelling in distributed inference and RL workloads. For Apple Silicon users, llama.cpp, Ollama, MLX, and llamafile are the strongest choices; for GPU-heavy deployments, vLLM and SGLang lead in throughput and feature completeness.
